# ADR 0002: Single-Source AI Agent Configuration

**Date:** 2026-06-19
**Status:** Accepted

## Context

The repo had a single `CLAUDE.md` at root holding all AI-agent guidance. We want the same guidance to drive other agentic tools (Cursor, OpenAI Codex, Factory, Jules, Gemini CLI, Zed, Amp, and so on), without maintaining one hand-written file per tool.

Two constraints shape the solution:

1. **No content drift.** The guidance should live in exactly one file. Duplicated copies across tool-specific files rot independently.
2. **Unity import hygiene.** This repo is the root of a UPM package (`package.json` at top). A consumer installing via git URL gets an immutable copy in `Library/PackageCache`; the local dev copy is mutable, so Unity generates `.meta` files into the working tree. Unity ignores only two path shapes: names starting with `.` and names ending with `~`. There is **no name-based way to exclude a dotless root file** like `AGENTS.md` from import.

`AGENTS.md` is the emerging cross-tool standard and is read natively by Cursor, Codex, Factory, Jules, Zed, and Amp.

## Decision

`AGENTS.md` is the **single source of truth**. The former `CLAUDE.md` content is migrated into it verbatim. Every other tool either reads `AGENTS.md` natively or points at it:

- **`CLAUDE.md`**: reduced to a one-line `@AGENTS.md` import (Claude Code has no setting to rename its context file, so the dotless file must stay).
- **`.gemini/settings.json`**: sets `contextFileName` to `AGENTS.md`. Lives in a dot-folder, so Unity never sees it.
- **Cursor / Codex / Factory / Jules / Zed / Amp**: read `AGENTS.md` natively. No extra files.

For Unity import hygiene, the two unavoidable dotless root files (`AGENTS.md` and `CLAUDE.md`) ship committed `.meta` files, matching the existing treatment of `README.md`, `CHANGELOG.md`, and `LICENSE.md`. The metas are generated by Unity (open the project once), not hand-written.

## Alternatives Considered

### A. Keep `CLAUDE.md` rich, generate the other files from it
A script or CI step copies content into `AGENTS.md` and others. Rejected: adds a build step and a drift window between source and generated copies. The `@import` / native-read model achieves zero drift with no tooling.

### B. Hide the agent files from Unity entirely
There is no mechanism to exclude a dotless root file by name. The only ways to hide a file are a `.`-prefix or `~`-suffix, both of which break the fixed filenames the tools require (`AGENTS.md`, `CLAUDE.md`). Rejected as impossible without dropping tool support. The files import as harmless `TextAsset`s instead, silenced by committed metas.

### C. First-class Copilot and Windsurf support
Both tools read their *own* file as literal context and cannot truly include an external `AGENTS.md`; a pointer stub there is only a soft instruction, and syncing full content would reintroduce drift. Rejected: dropped from scope to preserve strict single-source fidelity. Revisit if/when those tools gain native `AGENTS.md` support.

## Consequences

### Positive
- One file to edit. No drift across tools.
- Broad coverage (Cursor, Codex, Factory, Jules, Gemini, Zed, Amp, Claude) from a single source.
- `.gemini/` is invisible to Unity for free.

### Negative
- `AGENTS.md` and `CLAUDE.md` still import as `TextAsset`s in Unity. Accepted, identical to `README.md`; metas keep the import clean.
- Adding a new agent file requires regenerating its `.meta` in Unity before commit.

### Neutral
- Copilot and Windsurf are unsupported until they can include an external context file.

## Follow-Up Decisions Deferred

- First-class Copilot / Windsurf support if they adopt native `AGENTS.md` reading.
- Whether nested `AGENTS.md` files (per-subsystem) are worth adding for large folders like `Samples~`.
